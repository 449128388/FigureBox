# 2026-04-24 更新日志

## 资产服务层 Facade 模式重构（企业级架构）

### 问题背景
`asset_calculation_service.py` 和 `holding_analysis_service.py` 两个文件存在职责重叠，且部分方法已拆分到 `dashboard_service/assets_service/` 目录下的子服务中。需要进一步整合，采用 Facade 模式提供统一的向后兼容入口。

### 解决方案
创建 `assets_service.py` 作为 Facade 统一入口，整合 `AssetCalculationService` 和 `HoldingAnalysisService` 两个类，所有方法委托给相应的子服务实现。

### 服务架构重构

**重构前：**
- `app/services/asset_calculation_service.py` - 资产计算服务
- `app/services/holding_analysis_service.py` - 持仓分析服务

**重构后：**
- `app/services/dashboard_service/assets_service/assets_service.py` - Facade 统一入口（新增）
  - `AssetCalculationService` 类 - 资产计算 Facade
  - `HoldingAnalysisService` 类 - 持仓分析 Facade

**子服务清单：**
- `TotalAssetsCalculator` - 总资产计算
- `DailyChangeCalculator` - 日涨跌计算
- `PositionCalculator` - 仓位计算
- `DailyCacheService` - 每日市值缓存
- `MarketBenchmarkService` - 塑料指数、跑赢大盘计算
- `AssetDistributionService` - 风险分布、制造商分布等饼图统计
- `HoldingPositionService` - 持仓明细构建、总成本计算、状态标签确定
- `IndexService` - 上证指数、沪深300获取

### 统一入口方法

**AssetCalculationService 提供：**
- 总资产计算
- 总成本计算
- 日涨跌计算
- 市值缓存保存
- 仓位信息计算
- 塑料指数计算
- 跑赢大盘百分比计算
- 状态标签确定
- 持有天数计算
- 手办图片URL获取

**HoldingAnalysisService 提供：**
- 单个持仓明细构建
- 所有持仓明细构建
- 风险状态分布计算
- 制造商分布计算
- 持仓周期分布计算
- 仓位分层分布计算
- 所有分布数据综合分析

### 更新后的服务层次结构

```
backend/app/services/
├── __init__.py                           # 主服务层入口
└── dashboard_service/
    └── assets_service/
        ├── __init__.py                   # 资产服务模块入口
        ├── assets_service.py             # Facade 统一入口（新增）
        │   ├── AssetCalculationService   # 资产计算 Facade
        │   └── HoldingAnalysisService    # 持仓分析 Facade
        ├── asset_core_calculations.py    # 核心计算服务
        ├── asset_market_benchmark_service.py  # 市场基准服务
        ├── asset_distribution_service.py # 资产分布服务
        ├── holding_position_service.py   # 持仓定位服务
        └── index_service.py              # 指数服务
```

### 实现效果

- ✅ 原有 API 调用保持向后兼容，无需修改
- ✅ Facade 模式统一入口，代码结构清晰
- ✅ 业务逻辑拆分到独立子服务，符合单一职责原则
- ✅ 删除冗余文件，减少维护成本
- ✅ 提高代码可维护性和可测试性

## 价格更新服务合并到持仓定位服务

### 问题背景

- `get_figure_current_price` - 获取手办当前价格

- `calculate_price_update_impact` - 计算价格变更影响（原名 `calculate_impact`）
- `update_figure_price` - 更新手办价格

### 实现效果
- ✅ 持仓相关逻辑统一由 `HoldingPositionService` 管理
- ✅ 避免服务之间的重复代码
- ✅ 价格更新功能保持正常运行
- ✅ 减少服务文件数量，简化架构

### 删除的文件

- `app/services/price_update_service.py`

## 修改市场价支持多币种

### 问题背景
用户在修改手办市场价时，可能需要使用外币（如日元、美元等）进行定价，系统需要支持币种选择并自动转换为人民币进行计算和存储。

### 解决方案

**1. 前端实现**
在修改市场价弹窗中添加币种选择器：
- 支持币种：CNY（人民币）、JPY（日元）、USD（美元）、EUR（欧元）
- 默认为人民币（CNY）
- 选择非人民币币种时，显示汇率转换提示

**汇率配置（相对人民币）：**
- CNY: 1.0
- JPY: 1/23（1人民币 = 23日元）
- USD: 7.0（1美元 = 7人民币）
- EUR: 8.0（1欧元 = 8人民币）

**2. 后端实现**
API 接口支持接收币种参数，用于记录用户选择的原始币种。

**3. 计算逻辑**
- 用户输入外币价格
- 前端根据汇率转换为人民币
- 所有盈亏计算、总资产计算均基于人民币
- 最终结果以人民币展示

### 实现效果
- ✅ 支持多种币种输入市场价
- ✅ 自动汇率转换为人民币
- ✅ 所有计算结果统一为人民币
- ✅ 界面显示汇率转换提示
- ✅ 后端记录币种信息

## 持仓列表补仓功能

### 需求背景
用户需要在持仓列表中对已有持仓的手办进行补仓操作，增加库存数量并重新计算加权平均成本。

### 解决方案

**1. 前端实现**
- 在持仓列表的操作菜单中添加"补仓"选项
- 创建补仓对话框组件，包含：
  - 当前持仓信息展示（数量、成本价、市值）
  - 补仓数量和价格输入
  - 补仓后预览（加权平均成本、新库存）
- 使用 composable 抽离补仓业务逻辑

**2. 后端实现**
创建补仓服务，处理以下业务流程：
- 创建已完成的补仓订单（尾款管理）
- 创建 order_transactions 资金账记录
- 创建 asset_transactions 库存账记录
- 更新手办数量和加权平均成本
- 更新市值缓存（新买入部分当日不计入涨跌）

**3. 日涨跌特殊处理**
补仓的新买入部分当日不计入涨跌：
- 将新买入成本直接加到今日市值缓存中
- 日涨跌计算时，新买入部分贡献为0

### 数据变更关联
1. 尾款管理新增状态为"已完成"的订单
2. asset_transactions 和 order_transactions 新增记录
3. 手办数量增加，平均入手价格重新计算
4. 总资产增加，仓位发生变化
5. 市值缓存更新用于日涨跌计算

### 实现效果
- ✅ 持仓列表支持补仓操作
- ✅ 加权平均成本自动计算
- ✅ 所有相关数据自动更新
- ✅ 日涨跌计算正确处理新买入部分

### 部署更新

- 执行 `docker-compose up -d --build` 重新部署服务
