## 修复删除手办时的500错误

### 问题描述
删除手办 API (`DELETE /api/figures/{figure_id}`) 报错 500 Internal Server Error。

### 问题原因
删除手办时，没有先删除关联的 `asset_transactions` 记录，导致外键约束冲突。

**错误信息：**
```
ForeignKeyViolationError: 无法删除或更新父表 "figures" 的行，
因为存在外键约束 "asset_transactions_figure_id_fkey" 引用该行的子表 "asset_transactions"
```

### 修复方案

**文件：** `backend/app/services/figure_service.py`

```python
@staticmethod
def delete_figure(db: Session, figure_id: int) -> bool:
    db_figure = db.query(Figure).filter(Figure.id == figure_id).first()
    if not db_figure:
        return False
    
    # 检查是否有关联的订单
    associated_orders = db.query(Order).filter(
        Order.figure_id == figure_id
    ).first()
    
    if associated_orders:
        raise ValueError("无法删除有关联尾款的手办")
    
    # 【修复】先删除关联的资产交易记录
    from app.models.asset import AssetTransaction
    db.query(AssetTransaction).filter(
        AssetTransaction.figure_id == figure_id
    ).delete(synchronize_session=False)
    
    db.delete(db_figure)
    db.commit()
    return True
```

### 修复内容

1. 在删除手办之前，先删除 `asset_transactions` 表中 `figure_id` 关联的所有记录
2. 使用 `synchronize_session=False` 提高删除性能
3. 保持事务一致性（删除交易记录和手办在同一个事务中）

### 数据完整性说明

删除手办时同步删除关联的资产交易记录是合理的，因为：
- 资产交易记录是手办的附属数据
- 手办删除后，交易记录失去关联意义
- 保持数据库外键约束完整性

### 部署更新

- 执行 `docker-compose up -d --build` 重新部署服务，使修复生效

## 修复创建订单API的422错误

```python

    figure_id: int
    deposit: float
    balance: float
    due_date: date  # 必填字段

class OrderCreate(OrderBase):
    status: str
    # 继承自 OrderBase，due_date 仍然是必填
```

### 修复方案

**文件：** `backend/app/schemas/order.py`

```python
class OrderCreate(OrderBase):
    status: str
    # 【修复】已取消状态的订单不需要出荷日期，设置为可选
    due_date: date | None = None
```

### 修复内容

1. 在 `OrderCreate` 中覆盖 `due_date` 字段，设置为可选（`date | None = None`）
2. 保持 `OrderBase` 和 `OrderListItem` 中的 `due_date` 为必填（用于其他场景）
3. 与前端验证逻辑保持一致（已取消状态不需要出荷日期）

| 未支付 | ✅ | Schema 和前端都验证 |
| 已支付 | ✅ | Schema 和前端都验证 |

## 修复冲正交易扣减逻辑错误

### 问题描述
修改手办数量（减少）时，冲正交易错误地应用了FIFO（先进先出）卖出逻辑，导致资产归属混乱。

**错误表现：**
- 用户将数量从 3 → 1（减少2个）
- 系统扣减了**原始的1个**买入记录（remaining_quantity 从1变为0）
- 保留了**补录的2个**中的1个
- **资产归属完全颠倒！**

**图1（原始状态）：**
```
记录1: figure_id=24, trans_type='buy', quantity=1, remaining_quantity=1  (原始创建)
记录2: figure_id=24, trans_type='buy', quantity=2, remaining_quantity=2  (补录: 1→3)
```

**图2（错误状态）：**
```
记录1: figure_id=24, trans_type='buy', quantity=1, remaining_quantity=0  (被错误扣减！)
记录2: figure_id=24, trans_type='buy', quantity=2, remaining_quantity=1  (错误保留！)
记录3: figure_id=24, trans_type='adjust', quantity=-2  (冲正记录)
```

### 问题原因
冲正场景应该撤销**最近补录**的数量，而不是**最早原始**的数量。

**原逻辑（错误）：**
```python
# 先进先出 FIFO - 用于卖出场景
buy_transactions = db.query(...).order_by(
    AssetTransaction.transaction_date.asc()  # 先扣减最早的
).all()
```

### 修复方案

**文件：** `backend/app/services/asset_transaction_service.py`

```python
# 【修复】后进先出 LIFO - 用于冲正场景
buy_transactions = db.query(...).order_by(
    AssetTransaction.transaction_date.desc()  # 先扣减最近的（补录的）
).all()
```

### 修复后正确的数据表现

**场景：将手办数量从 3 → 1（减少2个）**

```
原始记录：
- 记录1: trans_type='buy', quantity=1, remaining_quantity=1  (原始创建 10:11:24)
- 记录2: trans_type='buy', quantity=2, remaining_quantity=2  (补录 10:11:39)

冲正后（修复后）：
- 记录1: trans_type='buy', quantity=1, remaining_quantity=1  ✓ 保留原始的
- 记录2: trans_type='buy', quantity=2, remaining_quantity=0  ✓ 撤销补录的
- 记录3: trans_type='adjust', quantity=-2  (冲正记录)
```

### 业务逻辑对比

| 场景 | 扣减策略 | 说明 |
|------|----------|------|
| **卖出操作** | FIFO（先进先出） | 卖出最早买入的资产 |
| **冲正操作** | LIFO（后进先出） | 撤销最近补录的数量 |

## 修复已取消状态订单的表单验证

### 问题描述
在尾款管理中，当添加订单时选择"尾款状态"为"已取消"，点击保存按钮没有反应。

**原验证逻辑（错误）：**
```javascript
const validateForm = () => {
  // 验证出荷日期 - 对所有状态都是必填
  if (!newOrder.value.due_date) {
    dueDateError.value = '请选择出荷日期'
    isValid = false
  }
}
```

### 修复方案

**文件：** `frontend/src/views/Orders/composables/useOrderManagement.js`

```javascript
const validateForm = () => {
  // 【修复】验证出荷日期 - 已取消状态的订单不需要填写出荷日期
  const isCancelled = newOrder.value.status === '已取消'
  if (!isCancelled && !newOrder.value.due_date) {
    dueDateError.value = '请选择出荷日期'
    isValid = false
  } else {
    dueDateError.value = ''
  }
}
```

### 修复内容

1. 在验证出荷日期前，先判断订单状态是否为"已取消"
2. 已取消状态的订单跳过出荷日期验证
3. 其他状态（未支付、已支付、已完成）仍需填写出荷日期

| 订单状态 | 出荷日期必填 | 说明 |

| 未支付 | ✅ | 需要预计出荷日期 |
| 已支付 | ✅ | 需要预计出荷日期 |
| 已完成 | ✅ | 需要实际出荷日期 |

class OrderTransaction(Base):
    """

    __tablename__ = "order_transactions"

    id = Column(Integer, primary_key=True, index=True)

    figure_id = Column(Integer, ForeignKey("figures.id"), nullable=False)

    currency = Column(String(10), default="CNY")

    notes = Column(String(255))

    figure = relationship("Figure")
    order = relationship("Order")

### 解决方案

#### 1. 数据库模型更新
**`backend/app/models/figure.py`**：
- 添加 `is_active` 字段（Boolean，默认 True）

- 为 `AssetTransaction` 添加 `is_active` 和 `deleted_at` 字段

#### 2. 业务逻辑更新

- **查询逻辑**：所有查询默认只返回 `is_active=True` 的记录
- **更新逻辑**：只更新激活状态的手办

- ✅ **数据恢复**：可通过修改 `is_active` 字段恢复误删数据

### 技术实现
- 使用 SQLAlchemy 的 `update()` 方法批量更新软删除字段
- 在查询时添加 `is_active == True` 条件过滤
- 保持数据库结构向后兼容

    AssetTransaction.order_id == order_id

    'is_active': False,
