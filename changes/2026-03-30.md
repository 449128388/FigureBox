## 1. 修复新增手办后排序问题

### 问题原因
在新增手办成功后，前端使用 `this.figures.push(response)` 将新手办直接添加到数组末尾，而不是重新从后端加载排序后的列表。这导致：
- 新增手办显示在列表最后
- 页面刷新后排序恢复正常
- 用户体验不一致

### 解决方案
在新增/编辑手办成功后，调用 `fetchFiguresWithSearch()` 重新加载列表数据，确保排序正确。

**文件**: `frontend/src/views/Figures.vue`

**修复前**:
```javascript
if (this.isEditing) {
  // 编辑模式
  await this.figureStore.updateFigure(this.currentEditFigureId, figureData)
} else {
  // 添加模式
  await this.figureStore.createFigure(figureData)
}

this.showAddForm = false
// 重置表单
this.resetForm()
```

**修复后**:
```javascript
if (this.isEditing) {
  // 编辑模式
  await this.figureStore.updateFigure(this.currentEditFigureId, figureData)
} else {
  // 添加模式
  await this.figureStore.createFigure(figureData)
}

this.showAddForm = false
// 重置表单
this.resetForm()

// 重新加载列表数据，确保排序正确
await this.fetchFiguresWithSearch()
```

### 功能效果
- ✅ 新增手办后立即重新加载列表
- ✅ 新手办按 id 降序显示在最前面
- ✅ 无需刷新页面即可看到正确排序
- ✅ 编辑手办后也能保持正确排序

### 技术要点
- **数据同步**：操作成功后重新从后端获取最新数据
- **排序一致**：确保前端显示与后端排序一致
- **用户体验**：操作后立即看到正确结果

### 用户体验优化
- **即时反馈**：新增手办后立即显示在正确位置
- **排序稳定**：无需手动刷新页面
- **操作流畅**：添加流程更加顺畅

## 4. 修复手办管理列表不展示标签的问题

### 问题描述
手办管理列表页面中，手办卡片没有展示标签信息。这是由于后端 API 使用了 SQLAlchemy 的懒加载（lazy loading）机制，导致标签关联数据没有被加载到响应中。

### 修复内容

#### 后端修改
**文件**: `backend/app/api/figures.py`

**1. 导入 joinedload**:
```python
from sqlalchemy.orm import Session, joinedload
```

**2. 在查询中使用 joinedload 预加载标签数据**:
```python
@router.get("/", response_model=list[FigureSchema])
def get_figures(
    skip: int = 0,
    limit: int = 100,
    name: str = None,
    purchase_type: str = None,
    purchase_date_start: str = None,
    purchase_date_end: str = None,
    tag_id: int = None,
    db: Session = Depends(get_db)
):
    """
    获取手办列表，支持搜索过滤
    """
    query = db.query(Figure)

    # ... 其他筛选条件 ...

    # 按 id 降序排序（最新的在前面）
    query = query.order_by(Figure.id.desc())

    # 使用 joinedload 预加载标签数据，避免 N+1 查询问题
    query = query.options(joinedload(Figure.tags))

    figures = query.offset(skip).limit(limit).all()
    return figures
```

### 技术说明
- **joinedload**: SQLAlchemy 的预加载选项，使用 JOIN 一次性加载关联数据
- **避免 N+1 问题**: 如果不使用预加载，访问每个手办的标签时都会触发一次查询
- **性能优化**: 通过单次查询获取所有需要的数据，减少数据库查询次数

### 修复效果
- ✅ 手办管理列表正确显示每个手办的标签
- ✅ 标签可以点击进行筛选
- ✅ 查询性能优化，避免 N+1 查询问题

## 5. 修复手办列表接口 500 错误

### 问题描述
`/api/figures/` GET 接口报错 500 Internal Server Error，错误信息为 `Out of sort memory, consider increasing server sort buffer size`。

### 问题原因
使用 `joinedload` 预加载标签数据时，与 `ORDER BY` 和 `LIMIT` 组合使用导致 MySQL 排序缓冲区不足。`joinedload` 使用 JOIN 查询，当数据量大时会产生大量重复行，增加排序内存压力。

### 修复内容

#### 后端修改
**文件**: `backend/app/api/figures.py`

**1. 将 joinedload 改为 selectinload**:
```python
from sqlalchemy.orm import Session, selectinload

# ...

@router.get("/", response_model=list[FigureSchema])
def get_figures(
    skip: int = 0,
    limit: int = 100,
    # ... 其他参数
    db: Session = Depends(get_db)
):
    query = db.query(Figure)
    
    # ... 筛选条件 ...
    
    # 按 id 降序排序（最新的在前面）
    query = query.order_by(Figure.id.desc())
    
    # 使用 selectinload 预加载标签数据，避免 N+1 查询问题和排序内存问题
    query = query.options(selectinload(Figure.tags))
    
    figures = query.offset(skip).limit(limit).all()
    return figures
```

### 技术说明
- **joinedload**: 使用 JOIN 一次性加载关联数据，但在分页和排序时可能导致内存问题
- **selectinload**: 使用额外的 IN 查询加载关联数据，避免 JOIN 带来的排序内存问题
- **性能对比**: 
  - `joinedload`: 单次查询，适合小数据量，大数据量时 JOIN 结果集膨胀
  - `selectinload`: 两次查询（主表+关联表），适合大数据量，避免内存问题

### 修复效果
- ✅ 手办列表接口不再报 500 错误
- ✅ 标签数据正确加载和显示
- ✅ 避免 MySQL 排序内存不足问题

## 8. 修复手办详情页面数据不显示问题

### 问题描述
点击手办进入详情页面后，页面没有任何数据展示。

### 问题原因
前端代码中使用了 `response.data` 来获取 API 返回的数据，但项目中的 axios 配置已经通过响应拦截器处理了数据返回：

```javascript
// axios/index.js 中的响应拦截器
instance.interceptors.response.use(
  response => {
    // ... token 续期逻辑 ...
    return response.data  // 直接返回 data
  },
  ...
)
```

这意味着 `axios.get()` 返回的已经是 `response.data`，而不是完整的 `response` 对象。代码中再访问 `.data` 会得到 `undefined`。

### 修复内容

#### 前端修改
**文件**: `frontend/src/views/FigureDetail.vue`

**修改数据获取逻辑**:
```javascript
// 修改前
const response = await axios.get(`/figures/${figureId}`)
this.figure = response.data

// 修改后
// axios拦截器已处理response.data，直接返回数据
const data = await axios.get(`/figures/${figureId}`)
this.figure = data
```

### 技术说明
- **Axios 拦截器**：在响应拦截器中直接返回 `response.data`，简化了调用方的代码
- **一致性**：项目中其他地方的 API 调用都是直接使用返回的数据，没有访问 `.data`
- **注意事项**：使用自定义 axios 实例时，需要了解拦截器的处理方式

### 修复效果
- ✅ 手办详情页面可以正常显示数据
- ✅ 所有字段（基本信息、规格信息、作者信息等）正确展示
- ✅ 关联的尾款信息正常显示

- 所有字段都使用 `v-if` 条件渲染，空对象导致所有条件都不满足
- 数据加载完成后才填充内容，造成从无到有的闪烁

```javascript
  }

```

**2. 修改数据获取方法，控制加载状态**:
```javascript
async fetchFigureDetail() {
  this.loading = true  // 开始加载
  const figureId = this.$route.params.id
  try {
    const data = await axios.get(`/figures/${figureId}`)
```

```html
    </div>
    
    <!-- 内容区域 -->

      <!-- 原有内容 -->
    </template>
  </div>
</template>
```

**4. 添加加载动画样式**:

## 11. 修复标签筛选功能 500 错误

### 问题描述
点击手办管理列表中手办的标签进行筛选时，接口 `http://localhost:25600/api/figures/?tag_id=2&skip=0&limit=15` 返回 500 Internal Server Error。

### 问题原因
错误信息：`(1038, 'Out of sort memory, consider increasing server sort buffer size')`

这是因为使用 `query.join(Figure.tags)` 进行 JOIN 操作后，再进行排序时 MySQL 排序缓冲区不足，导致查询失败。

### 修复内容

#### 后端修改
**文件**: `backend/app/api/figures.py`

**1. 导入 figure_tag 表**:
```python
from app.models.tag import Tag, figure_tag
```

**2. 优化标签筛选逻辑**:
```python
# 按标签ID筛选
if tag_id:
    # 先执行子查询获取包含指定标签的手办ID列表
    # 避免在主查询中使用子查询导致的排序内存问题
    figure_ids = db.query(figure_tag.c.figure_id).filter(figure_tag.c.tag_id == tag_id).all()
    # 提取ID值
    figure_id_list = [id_tuple[0] for id_tuple in figure_ids]
    # 如果没有符合条件的手办，直接返回空列表
    if not figure_id_list:
        return []
    # 然后根据ID列表筛选
    query = query.filter(Figure.id.in_(figure_id_list))
```

### 技术说明
- **分步查询方案**：先执行简单查询获取包含指定标签的手办ID列表，再根据ID列表进行筛选
- **避免复杂查询**：这种方式避免了在单个查询中同时进行JOIN和排序，减少了内存消耗
- **性能优化**：分步查询比复杂的子查询更轻量，特别是在处理多对多关系时

### 修复效果
- ✅ 标签筛选功能恢复正常
- ✅ 不再出现 500 Internal Server Error
- ✅ 筛选性能得到提升
