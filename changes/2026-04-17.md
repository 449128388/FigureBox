## 优化入手价格计算逻辑（支持多批次补仓）

### 问题背景
原系统在手办列表和表单中直接显示 `purchase_price` 字段，但这种方式无法解决多批次补仓场景的问题：
- 第一次购买：1个，价格 ¥800
- 第二次补仓：2个，价格 ¥750
- 实际平均入手价格应该是：(800×1 + 750×2) / 3 = ¥766.67

### 解决方案

**1. 后端模型优化（`backend/app/models/figure.py`）**：
- 添加 `orders` 关联关系（一对多）
- 新增 `calculate_average_purchase_price()` 方法，根据订单计算加权平均价格

```python
class Figure(Base):
    # ... 其他字段 ...
    orders = relationship("Order", back_populates="figure")

    def calculate_average_purchase_price(self):
        """根据关联订单计算加权平均入手价格"""
        if not self.orders:
            return 0

        total_amount = 0
        total_quantity = 0

        for order in self.orders:
            order_amount = (order.deposit or 0) + (order.balance or 0)
            order_quantity = getattr(order, 'quantity', 1) or 1
            total_amount += order_amount
            total_quantity += order_quantity

        return total_amount / total_quantity if total_quantity > 0 else 0
```

**2. 后端 API 优化（`backend/app/api/figures.py`）**：
- 修改 `get_figure` 接口，加载关联订单并计算平均入手价格
- 新增 `average_purchase_price` 字段返回给前端

**3. Schema 更新（`backend/app/schemas/figure.py`）**：
- `Figure` schema 新增 `average_purchase_price` 字段

**4. 前端列表优化（`frontend/src/views/Figures/components/FigureItem.vue`）**：
- 屏蔽入手价格显示（避免展示静态字段）

**5. 前端表单优化（`frontend/src/views/Figures/components/form/FormBasicTab.vue`）**：
- 入手价格字段改为只读显示
- 显示 "自动计算" 标签
- 添加提示说明：根据关联订单的（定金+尾款）自动计算加权平均价格

### 计算逻辑
```
平均入手价格 = Σ(订单定金 + 订单尾款) / Σ(订单数量)

示例：
- 订单1：1个，定金¥200，尾款¥600 → 总金额¥800
- 订单2：2个，定金¥0，尾款¥1500 → 总金额¥1500
- 平均价格 = (800 + 1500) / (1 + 2) = ¥766.67
```

### 优化效果
- ✅ 支持多批次补仓场景的平均价格计算
- ✅ 列表中不再显示静态入手价格（避免误导）
- ✅ 详情/编辑页面显示自动计算的平均入手价格
- ✅ 无订单时默认显示 0
- ✅ 计算逻辑透明，用户可理解

```python
def get_figure_orders_count(figure_id: int, db: Session = Depends(get_db)):
    """获取手办关联的订单数量"""
    from app.models.order import Order
    order_count = db.query(Order).filter(Order.figure_id == figure_id).count()
```

```javascript
const showDeleteConfirmDialog = ref(false)

const figureToDeleteOrderCount = ref(0)

// 打开删除确认对话框
const openDeleteConfirmDialog = async (figure) => {
  figureToDelete.value = figure
  showDeleteConfirmDialog.value = true
  // 获取关联订单数量
  const response = await axios.get(`/figures/${figure.id}/orders/count`)
  figureToDeleteOrderCount.value = response.count || 0

// 取消删除

  showDeleteConfirmDialog.value = false
  figureToDelete.value = null
  figureToDeleteOrderCount.value = 0
}

// 确认删除
const confirmDelete = async () => {
  if (!figureToDelete.value) return
  await figureStore.deleteFigure(figureToDelete.value.id)
  // ... 清理状态
}

```

## 订单管理实现软删除功能

### 需求背景
为了提高数据安全性和审计能力，订单删除不再物理删除记录，而是采用软删除机制，仅标记订单为已删除状态并记录删除时间。

### 后端修改

**1. Order 模型新增软删除字段**
- `is_active` (Integer, default=1) - 是否激活：1=正常，0=已删除
- `deleted_at` (DateTime, nullable=True) - 删除时间（软删除标记）

**2. 订单删除 API 改为软删除**
- 不再使用 `db.delete()` 物理删除
- 改为设置 `is_active=0` 和 `deleted_at=当前时间`
- 同时软删除关联的资产交易记录和资金流水记录

**3. 订单查询 API 更新**
- 所有查询订单的接口都添加 `is_active=1` 过滤条件
- 只返回未软删除的订单
- 涉及的接口：
  - `GET /api/orders/` - 订单列表
  - `GET /api/orders/{id}/` - 订单详情
  - `PUT /api/orders/{id}/` - 更新订单
  - `GET /api/orders/unpaid-balance/` - 未支付尾款总额
  - `POST /api/orders/` - 创建订单时检查订单数量

**4. 手办删除逻辑更新**
- 检查手办关联的订单时，只检查未软删除的订单（`is_active=1`）
- 如果存在未软删除的关联订单，禁止删除手办
- 提示信息更新为："无法删除有关联尾款的手办，请先删除或软删除所有关联订单"

**5. 手办订单数量统计 API 更新**
- `GET /api/figures/{id}/orders/count` 只统计未软删除的订单

### 修改文件
- `backend/app/models/order.py` - 新增软删除字段
- `backend/app/api/orders.py` - 更新订单相关 API
- `backend/app/api/figures.py` - 更新订单数量统计
- `backend/app/services/figure_service.py` - 更新手办删除逻辑

### 代码示例

**订单模型软删除字段**:
```python
# 软删除标记
is_active = Column(Integer, default=1)  # 是否激活：1=正常，0=已删除
deleted_at = Column(DateTime, nullable=True)  # 删除时间（软删除标记）
```

**订单软删除实现**:
```python
@router.delete("/{order_id}/")
def delete_order(order_id: int, ...):
    # ... 权限检查 ...
    
    # 软删除关联的资产交易记录
    db.query(AssetTransaction).filter(
        AssetTransaction.order_id == order_id
    ).update({
        'is_active': False,
        'deleted_at': datetime.now(),
        'order_id': None
    })
    
    # 软删除关联的资金流水记录
    db.query(OrderTransaction).filter(...).update({...})
    
    # 软删除订单本身
    db_order.is_active = 0
    db_order.deleted_at = datetime.now()
    db.commit()
```

**手办删除时检查关联订单**:
```python
# 检查是否有关联的未软删除订单（is_active=1）
active_orders = db.query(Order).filter(
    Order.figure_id == figure_id,
    Order.is_active == 1
).first()

if active_orders:
    raise ValueError("无法删除有关联尾款的手办，请先删除或软删除所有关联订单")
```

## 修复创建订单时交易记录逻辑

### 需求背景
创建订单时发现两个问题：
1. **资金账缺失**：创建订单时只创建了库存账记录，没有创建资金流水记录（order_transactions）
2. **库存账重复**：创建订单时会新增一条 asset_transactions 记录，导致手办数量虚增（如从5个变成6个）

### 后端修改

**1. AssetTransactionService 重构**
- 将 `create_buy_transaction_from_order` 方法重命名为 `link_order_to_existing_transaction`
- 改为**补录凭证模式**：不再新增交易记录，而是将订单关联到原有的库存记录
- 查找逻辑：
  1. 查找该手办下无订单关联的买入记录（`order_id IS NULL`）
  2. 如果找到，更新 `order_id` 并添加补录备注
  3. 如果没找到，返回 None

**2. 创建订单 API 更新**
在 `POST /api/orders/` 中同时创建两种交易记录：
- **库存账（asset_transactions）**：通过 `link_order_to_existing_transaction` 将订单关联到原有记录
- **资金账（order_transactions）**：通过 `OrderTransactionService.create_buy_transaction` 创建资金流水

**代码示例**:
```python
# 创建订单时的交易记录处理
try:
    # 1. 将订单关联到现有的库存记录（补录凭证模式）
    AssetTransactionService.link_order_to_existing_transaction(
        db=db,
        user_id=current_user.id,
        figure_id=order.figure_id,
        order=db_order
    )

    # 2. 创建资金流水记录（资金账）
    OrderTransactionService.create_buy_transaction(
        db=db,
        user_id=current_user.id,
        figure_id=order.figure_id,
        order_id=db_order.id,
        quantity=1,
        unit_price=total_price,
        total_amount=total_price,
        notes=f"订单 #{db_order.id} 资金流水"
    )

    db.commit()
except Exception as e:
    db.rollback()
    print(f"创建交易记录失败: {e}")
```

### 修改文件
- `backend/app/services/asset_transaction_service.py` - 重构为补录凭证模式
- `backend/app/api/orders.py` - 更新创建订单逻辑，同时创建两种交易记录

### 业务逻辑对比

| 场景 | 修改前 | 修改后 |
|------|--------|--------|
| 创建手办 | 创建 asset_transactions（库存账） | 创建 asset_transactions（库存账） |
| 创建订单 | 新增 asset_transactions（数量虚增） | 关联订单到原有库存记录（补录凭证） |
| 资金流水 | ❌ 未创建 | ✅ 创建 order_transactions（资金账） |

### 优化效果
- ✅ 创建订单时不再虚增手办数量
- ✅ 库存账采用补录凭证模式，订单关联到原有记录
- ✅ 创建订单时同时创建资金流水记录
- ✅ 库存账和资金账完全解耦，各负其责
