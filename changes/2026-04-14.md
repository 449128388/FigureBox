## Bug修复

### 导入手办功能修复

**问题描述：**
在手办管理页面中使用导入手办功能时，前端页面提示"导入失败"，但后端接口返回 200 状态码。

**问题原因：**
`ImportFiguresDialog.vue` 组件中使用了 `await emit('import', previewData.value)` 来等待导入结果，但 Vue 的 `$emit` 是同步的，不会返回 Promise 或等待父组件处理完成，导致导入结果无法正确获取。

**修复内容：**
- `frontend/src/views/Figures/components/ImportFiguresDialog.vue`
  - 修改 `handleImport` 方法，使用 Promise 包装 emit，通过回调函数机制等待父组件处理完成
  
- `frontend/src/views/Figures.vue`
  - 修改 `handleImport` 方法，接收回调函数参数并在处理完成后调用回调返回结果

**修复后行为：**
- 导入手办功能正常工作
- 导入成功时显示成功状态和导入统计
- 导入失败时显示错误信息

### 导入功能修复（续）

**问题描述：**
修复导入功能后，前端仍然提示"导入失败"，但后端返回 200 状态码。

**问题原因：**
- 前端使用了错误的字段名：`imported_count` 和 `orders_count`，但后端返回的是 `imported_figures` 和 `imported_orders`
- 后端在跳过已存在的手办时，仍然增加了 `updated_figures` 计数，这是不正确的

**修复内容：**
- `frontend/src/views/Figures/composables/useImportFigures.js`
  - 修正字段名，使用 `imported_figures` 和 `imported_orders`
  
- `backend/app/services/figure_import_service.py`
  - 移除 `updated_figures` 计数逻辑，因为现在不会更新已存在的手办

### 导入功能修复（最终）

**问题描述：**
前端提示 "Cannot read properties of undefined (reading 'success')"，但后端返回 200 状态码。

**问题原因：**

**修复内容：**
- `frontend/src/views/Figures/composables/useImportFigures.js`
  - 移除 `.data` 访问，因为 axios 响应拦截器已经返回了数据
  - 修正字段名，使用 `imported_count` 和 `orders_count` 与后端保持一致

### 导入逻辑优化（续）

**问题描述：**
导入 `figures_2026-04-13.json` 文件时，显示 "成功导入 0 个新手办，更新 0 个手办，导入 0 个订单"，但文件中包含 3 个手办和 6 个订单。

**修复内容：**
- `backend/app/services/figure_import_service.py`
  - 修改 `import_figure` 方法：总是创建新的手办，不检查是否存在
  - 修改 `import_orders` 方法：总是创建新的订单，不检查是否存在

**修复后行为：**
- 导入时总是创建新的手办和订单记录
- 不会因为已存在同名手办或相同ID订单而跳过导入

### 导入逻辑优化

**修改内容：**
修改 `backend/app/services/figure_import_service.py` 中的导入逻辑：

1. **手办导入逻辑 (`import_figure` 方法)**：
   - 如果手办名称已存在，直接跳过，不进行任何更新
   - 只创建新的手办记录

2. **订单导入逻辑 (`import_orders` 方法)**：
   - 如果订单ID已存在，直接跳过，不进行任何更新
   - 只为新手办创建新的订单记录

**修改原因：**
- 避免导入时意外覆盖现有手办和订单数据
- 确保导入操作只进行增量添加，不影响已有数据
- 提高数据安全性，防止误操作导致数据丢失

```

                → order_id = NULL
                → quantity = 手办数量
                → total_amount = 入手价格 × 数量
```

#### 场景2：创建手办 + 创建订单
```
用户创建手办 → 自动创建 AssetTransaction（买入，order_id=NULL）

                → 如果存在：更新 order_id
                → 如果不存在：创建新的交易记录
```

#### 场景3：补仓操作
```
用户点击补仓 → 填写价格、数量 → 创建新的 AssetTransaction（买入）
                → 重新计算平均成本
```

#### 场景4：卖出操作
```
用户点击卖出 → 填写价格、数量 → 创建 AssetTransaction（卖出）
                → 先进先出扣减买入记录的 remaining_quantity
                → 计算已实现盈亏
```

## 修复手办导入时缺少资产交易记录

### 问题描述
手办导入功能（JSON导入）时，`asset_transactions` 表中缺少对应的交易记录。

**原因分析：**
1. `FigureImportService.import_figure()` 直接创建 `Figure` 对象，**没有调用 `FigureService.create_figure()`**
2. `FigureImportService.import_orders()` 直接创建 `Order` 对象，**没有调用 `AssetTransactionService.create_buy_transaction_from_order()`**

### 修复方案

**1. 修复手办导入逻辑：**
```python
# 【修复前】直接创建 Figure 对象
figure = Figure()
figure.name = figure_data.get('name', '')
# ... 设置其他字段
db.add(figure)
db.flush()

# 【修复后】使用 FigureService 创建手办
processed_data = {
    'name': figure_data.get('name', ''),
    # ... 其他字段
}
figure = FigureService.create_figure(db, processed_data, user_id=user_id)
# 自动创建 asset_transactions 记录
```

**2. 修复订单导入逻辑：**
```python
# 【修复前】直接创建 Order 对象
order = Order()
order.figure_id = figure.id
# ... 设置其他字段
db.add(order)

# 【修复后】创建订单后自动创建交易记录
db.add(order)
db.flush()

AssetTransactionService.create_buy_transaction_from_order(
    db=db,
    user_id=user_id,
    figure_id=figure.id,
    order=order,
    quantity=1
)
```

### 修复内容

## 修复塑料手办指数计算错误

### 问题描述
塑料手办指数出现不合理的数据（如 1000000），与原始需求"市值加权复权指数"不符。

### 原始需求
- **基准日**：购买第一只手办的日期（开户首日）
- **基准日指数**：1000点
- **基准日总市值**：当日所有持仓手办的入手价格（初始投入成本）
- **公式**：塑料指数 = 基准日指数 × (当前总市值 / 基准日总市值)

### 问题原因
1. **没有手办时的处理错误**：当没有手办时，`base_total_value` 被设为 1，导致指数计算异常
2. **除以极小值的隐患**：当成本数据缺失时，使用 1 作为基准会导致指数放大

### 修复内容
**文件：** `backend/app/services/asset_calculation_service.py`

```python
# 修复前
if purchase_dates:
    base_date = min(purchase_dates)
    base_total_value = sum(
        (fig.purchase_price or 0) * (fig.quantity or 1) 
        for fig in figures
    )
else:
    base_date = date.today()
    base_total_value = 1  # 错误：导致指数异常

if base_total_value > 0:
    plastic_index = round(
        cls.BASE_INDEX * (total_assets / base_total_value), 2
    )
else:
    plastic_index = cls.BASE_INDEX

# 修复后
if not purchase_dates:
    # 没有手办时，返回基准指数1000和当前日期
    return cls.BASE_INDEX, date.today()

base_date = min(purchase_dates)

# 基准日总市值 = 基准日当天所有持仓手办的入手价格总和
base_total_value = sum(
    (fig.purchase_price or 0) * (fig.quantity or 1) 
    for fig in figures
)

# 如果没有成本数据，使用当前总资产作为基准
if base_total_value <= 0:
    base_total_value = total_assets if total_assets > 0 else cls.BASE_INDEX

# 计算塑料指数
plastic_index = round(
    cls.BASE_INDEX * (total_assets / base_total_value), 2
)
```

### 修复效果

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 没有手办 | 可能显示 1000000 | 显示 1000（基准指数） |
| 有手办无成本数据 | 可能显示异常大值 | 使用当前总资产作为基准 |
| 正常情况 | 可能计算错误 | 正确反映资产配置能力 |

## 修复资产交易记录业务逻辑

### 问题描述
当前创建订单时的资产交易记录逻辑存在严重问题：

1. **丢失原始数据**：创建订单时更新手办创建的交易记录，导致丢失手办入库时间和原始价格
2. **数量管理混乱**：多个订单时，原有记录的 `remaining_quantity` 没有正确变化，新记录独立计算

**错误的数据表现：**
```
记录1: figure_id=16, order_id=NULL, price=800, quantity=1, remaining_quantity=1  (手办创建)
记录2: figure_id=18, order_id=19, price=888, quantity=5, remaining_quantity=5    (订单#19)
记录3: figure_id=18, order_id=20, price=800, quantity=1, remaining_quantity=1    (订单#20)
```

### 问题分析

**原逻辑：**
```python
# 查找无订单关联的交易记录
existing = 查询(order_id=NULL的交易记录)
if existing:
    # 错误：更新原有记录
    existing.order_id = 新订单ID
    existing.price = 新价格
    # 丢失原始数据！
```

**问题：**
- 手办创建记录被修改，丢失入库时间
- 每个订单应该独立成记录，但原逻辑会覆盖
- 总持仓计算需要累加所有记录的 remaining_quantity

### 修复方案

**新逻辑：**
```python
# 【修复】始终创建新的交易记录，不更新现有记录
# 保留手办创建的原始记录完整性
# 每个订单对应一条独立的买入记录
transaction = AssetTransaction(
    user_id=user_id,
    figure_id=figure_id,
    order_id=order.id,  # 关联订单
    transaction_type="buy",
    price=total_price,
    quantity=quantity,
    total_amount=total_price * quantity,
    remaining_quantity=quantity,  # 新订单的剩余数量
    notes=f"从订单 #{order.id} 创建"
)
```

### 修复后正确的数据表现

```
记录1: figure_id=16, order_id=NULL, price=800, quantity=1, remaining_quantity=1  (手办创建-保留)
记录2: figure_id=18, order_id=19, price=888, quantity=5, remaining_quantity=5    (订单#19-独立)
记录3: figure_id=18, order_id=20, price=800, quantity=1, remaining_quantity=1    (订单#20-独立)
```

**总持仓计算：**
- figure_id=18 的总持仓 = 5 + 1 = 6
- 卖出时按先进先出扣减各记录的 remaining_quantity

### 业务逻辑规范

| 场景 | 行为 | 说明 |
|------|------|------|
| 创建手办 | 创建 AssetTransaction（买入，order_id=NULL） | 保留原始入库时间和价格 |
| 创建订单 | **新增** AssetTransaction（买入，order_id=订单ID） | 不修改手办记录，独立成记录 |
| 补仓 | 新增 AssetTransaction（买入） | remaining_quantity = 补仓数量 |
| 卖出 | 新增 AssetTransaction（卖出） | 扣减买入记录的 remaining_quantity |

### 部署更新

- 执行 `docker-compose up -d --build` 重新部署服务，使修复生效
