## 修复订单资金流水币种计算错误

### 问题背景
`order_transactions` 表中的 `unit_price` 和 `total_amount` 字段计算未考虑定金和尾款的币种差异，导致多币种订单的资金记录金额错误。

### 问题分析
**问题场景示例：**
- 订单定金：1000 JPY（约 43 CNY）
- 订单尾款：500 CNY
- **错误计算：** 1000 + 500 = 1500（直接数值相加）
- **正确计算：** 43 + 500 = 543（转换为人民币后相加）

**问题代码位置 1：** `backend/app/services/order_service/order_crud_service.py` 第 88-96 行
```python
# 原代码直接相加，未考虑币种转换
total_price = db_order.deposit + db_order.balance  # 错误！
OrderTransactionService.create_buy_transaction(
    ...
    unit_price=total_price,
    total_amount=total_price,
    ...
)
```

**问题代码位置 2：** `backend/app/services/order_transaction_service.py` 第 291-292 行
```python
# 原代码直接相加，未考虑币种转换
total_amount = (order.deposit or 0) + (order.balance or 0)  # 错误！
```

**问题影响：**
- 多币种订单的资金流水金额计算错误
- 影响资金统计和财务报表的准确性
- 导致资产价值评估偏差

### 修复方案
使用 `FigurePriceService.calculate_order_amount_cny()` 方法统一处理币种转换。

**修改文件 1：** `backend/app/services/order_service/order_crud_service.py`

**修改前：**
```python
total_price = db_order.deposit + db_order.balance
```

**修改后：**
```python
total_price = FigurePriceService.calculate_order_amount_cny(
    deposit=db_order.deposit,
    deposit_currency=db_order.deposit_currency,
    balance=db_order.balance,
    balance_currency=db_order.balance_currency
)
```

**修改文件 2：** `backend/app/services/order_transaction_service.py`

**修改前：**
```python
total_amount = (order.deposit or 0) + (order.balance or 0)
```

**修改后：**
```python
total_amount = FigurePriceService.calculate_order_amount_cny(
    deposit=order.deposit,
    deposit_currency=order.deposit_currency,
    balance=order.balance,
    balance_currency=order.balance_currency
)
```

**支持的币种汇率：**
- CNY（人民币）：1.0（基准）
- JPY（日元）：1人民币 = 23日元
- USD（美元）：1美元 = 7人民币
- EUR（欧元）：1欧元 = 8人民币

### 修复效果

## 修复订单与库存关联缺失问题（悬空订单）

### 问题背景
订单表存在订单#3（figure_id=2），但 `asset_transactions` 中无 `order_id=3` 的记录，导致订单-库存脱节，形成悬空订单。

### 问题分析
**根因：** 订单创建时未自动检索同 `figure_id` 下 `order_id IS NULL` **且** `remaining_quantity > 0` 的孤儿库存进行关联。

**问题代码位置：** `backend/app/services/asset_transaction_service.py` 第 92-98 行

```python
# 原代码只检查 order_id IS NULL，未检查 remaining_quantity
existing_transaction = db.query(AssetTransaction).filter(
    AssetTransaction.user_id == user_id,
    AssetTransaction.figure_id == figure_id,
    AssetTransaction.transaction_type == "buy",
    AssetTransaction.order_id.is_(None)  # 缺少 remaining_quantity > 0 检查
).first()
```

**问题影响：**
- 库存已完全卖出（`remaining_quantity=0`）的交易记录仍会被关联到新订单
- 导致订单存在但关联的库存记录已无实际库存，形成"悬空订单"
- 影响库存准确性和资产管理

### 修复方案
在 `link_order_to_existing_transaction` 方法中添加 `remaining_quantity > 0` 条件过滤。

**修改文件：** `backend/app/services/asset_transaction_service.py`

**修改前：**
```python
# 查找该手办下无订单关联的买入记录
existing_transaction = db.query(AssetTransaction).filter(
    AssetTransaction.user_id == user_id,
    AssetTransaction.figure_id == figure_id,
    AssetTransaction.transaction_type == "buy",
    AssetTransaction.order_id.is_(None)
).first()
```

**修改后：**
```python
# 查找该手办下无订单关联且有剩余库存的买入记录
existing_transaction = db.query(AssetTransaction).filter(
    AssetTransaction.user_id == user_id,
    AssetTransaction.figure_id == figure_id,
    AssetTransaction.transaction_type == "buy",
    AssetTransaction.order_id.is_(None),
    AssetTransaction.remaining_quantity > 0  # 新增：确保库存有剩余
).first()
```

### 修复效果
- 订单创建时只关联有实际库存（`remaining_quantity > 0`）的交易记录
- 防止悬空订单产生
- 确保订单-库存关联的准确性
- 提高资产管理的可靠性

## 重构库存账创建逻辑（订单驱动模式）

### 问题背景
原逻辑存在以下问题：
1. 创建手办时自动创建 `asset_transactions` 库存账记录
2. 创建订单时通过 `link_order_to_existing_transaction` 关联已有库存记录
3. 用户创建了3条订单，但 `asset_transactions` 中只有2条记录带有 `order_id`，第3个订单漏关联

### 问题分析
**根本原因：**
- 手办创建时只创建了一条库存记录（quantity = 手办数量）
- 订单创建时只能关联到这条已存在的记录
- 当订单数量超过1个时，后续订单无法找到可关联的库存记录（`order_id IS NULL` 的记录只有一条）

**原流程：**
```
创建手办 → 创建1条asset_transactions（quantity=手办数量, order_id=NULL）
创建订单1 → 关联到asset_transactions（设置order_id=1）
创建订单2 → 关联到asset_transactions（设置order_id=2）？？？
```
实际上第一个订单关联后，`order_id` 已被占用，后续订单无法再关联。

### 修复方案
改为**订单驱动模式**：
1. **删除**创建手办时的 `asset_transactions` 创建逻辑
2. **修改**为创建订单时**新建** `asset_transactions` 记录
3. 每个订单对应一条独立的库存记录

**修改文件 1：** `backend/app/services/figure_service/figure_crud_service.py`

**删除内容：**
```python

# 【修复】无论入手价格是否为0，都创建交易记录，保证数据完整性
```

## 修复编辑手办数量创建冲正记录的问题

### 问题背景
在采用订单驱动模式后，编辑手办数量时仍会创建数量调整冲正交易记录，这导致：
1. 手办数量应该由订单数量决定，直接编辑数量造成数据不一致
2. 创建多余的冲正交易记录，干扰正常的库存账

### 问题分析
**原逻辑：**
```python
# 编辑手办时，如果数量变化，创建冲正交易
if 'quantity' in figure_data and new_quantity != original_quantity:
    AssetTransactionService.create_quantity_adjustment_transaction(...)
```

**问题：**
- 订单驱动模式下，手办数量应该等于关联订单数量
- 直接编辑手办数量并创建冲正记录会造成：
  - 手办数量与订单数量不匹配
  - 产生多余的 `asset_transactions` 冲正记录
  - 库存数据混乱

### 修复方案
移除编辑手办时的冲正交易创建逻辑。

**修改文件：** `backend/app/api/figures.py`

**删除内容：**
```python
# 处理冲正交易
try:
    # 检查数量变化
    new_quantity = db_figure.quantity or 1
    if 'quantity' in figure_data and new_quantity != original_quantity:
        quantity_change = new_quantity - original_quantity
        adjustment_price = db_figure.average_purchase_price or original_price or 0
        
        AssetTransactionService.create_quantity_adjustment_transaction(
            db=db,
            user_id=current_user.id,
            figure_id=figure_id,
            quantity_change=quantity_change,
            price=adjustment_price,
            original_quantity=original_quantity,
            new_quantity=new_quantity
        )
        db.commit()
except Exception as e:
    db.rollback()
    print(f"创建冲正交易记录失败: {e}")
```

**更新文档说明：**
```python
"""
注意：
- 手办数量由关联订单数量决定，直接编辑数量不再创建冲正交易
- 价格调整通过订单均价计算自动更新
"""
```

### 修复效果
- ✅ 编辑手办数量不再创建冲正交易记录
- ✅ 手办数量与订单数量保持一致
- ✅ 库存账只记录订单相关的交易
- ✅ 数据逻辑更加清晰一致

## 修复已取消订单的交易记录缺陷

### 问题背景
对于状态为"已取消"的订单，`order_transactions` 和 `asset_transactions` 中存在数据缺陷：
1. 不应该记录完整的购买金额，而应该只记录定金
2. 不应该记录数量，因为订单已取消
3. 交易类型应该为"deposit"而不是"buy"

### 问题分析
**原逻辑问题：**
- 已取消订单创建了与普通订单相同的交易记录
- 记录了完整的金额（定金+尾款）
- 记录了数量=1
- 交易类型为"buy"

**正确逻辑：**
- 已取消订单只损失定金，不应该记录尾款
- 订单已取消，数量为0
- 交易类型应该为"deposit"（定金）

### 修复方案
根据订单状态分别处理交易记录的创建。

**修改文件 1：** `backend/app/services/figure_service/figure_price_service.py`

**新增方法：**


```python
            Order.status == "未支付",
            Order.user_id == current_user.id,
            Order.is_active == 1
        ).scalar()
    return {"total_unpaid_balance": float(total_balance) if total_balance else 0.0}
```

**问题：**
- 直接使用 `func.sum(Order.balance)` 对尾款金额求和
- 忽略了 `balance_currency` 币种字段
- 导致不同币种的金额被直接相加，结果错误

```python
@staticmethod
def get_unpaid_balance(db: Session, current_user: User) -> dict:
    # 获取所有未支付订单
    if current_user.is_admin:
        unpaid_orders = db.query(Order).filter(
            Order.status == "未支付",
            Order.is_active == 1
        ).all()
    else:
        unpaid_orders = db.query(Order).filter(
            Order.status == "未支付",
            Order.user_id == current_user.id,
            Order.is_active == 1
        ).all()

    # 将所有尾款按币种转换为人民币后求和
    total_balance_cny = 0.0
    for order in unpaid_orders:
        balance_cny = FigurePriceService.convert_to_cny(
            order.balance or 0,
            order.balance_currency or 'CNY'
        )
        total_balance_cny += balance_cny

    return {"total_unpaid_balance": total_balance_cny}
```

### 修复内容
1. **导入货币转换服务：** 引入 `FigurePriceService` 进行币种转换
2. **获取订单列表：** 改为查询所有未支付订单对象（而非直接求和）
3. **币种转换求和：** 对每个订单的尾款金额按币种转换为人民币后累加
4. **支持所有币种：** 支持 CNY、JPY、USD、EUR 等币种的正确统计

### 修复效果
- ✅ 待补款统计现在支持所有币种（人民币、日元、美元、欧元等）
- ✅ 不同币种的尾款金额会按汇率正确转换为人民币后求和

## 订单资金变更追踪功能实现

### 问题背景
目前资金账（`order_transactions`）对于手办的定金、尾款、币种发生变化时缺少对应的记录，只记录了订单创建时的原始数据。当用户编辑订单修改金额或币种时，无法追踪资金变动历史。

### 解决方案
采用**方案一：扩展资金流水表**，新增变更追踪字段和独立服务模块。

### 数据库模型扩展

**文件：** `backend/app/models/asset.py`

**新增字段：**
```python

transaction_subtype = Column(String(50), nullable=True)  # 交易子类型
parent_transaction_id = Column(Integer, ForeignKey("order_transactions.id"), nullable=True)  # 父交易ID
change_reason = Column(String(255), nullable=True)  # 变更原因
previous_amount = Column(Float, nullable=True)  # 变更前金额
current_amount = Column(Float, nullable=True)  # 变更后金额
changed_field = Column(String(50), nullable=True)  # 变更字段（deposit/balance）
```

### 新增服务模块

**文件：** `backend/app/services/order_transaction_service/transaction_change_service.py`

**核心功能：**
1. **变更检测服务** (`detect_and_record_changes`)
   - 检测定金金额/币种变更
   - 检测尾款金额/币种变更
   - 自动判断变更类型（追加、退款、币种变更）

2. **变更类型定义：**
   - `initial`：初始创建
   - `supplement`：追加金额

   - 金额增加 → 创建追加记录（资金流出）
