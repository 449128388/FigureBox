## 修复手办详情页居中问题

### 问题描述
手办详情页面中的 `figure-detail-container` 容器未正确居中显示，导致页面布局偏左。

### 问题原因
原 CSS 样式使用 `margin-left: 50px; margin-right: 50px;`，这种写法无法实现真正的水平居中。

### 修复方案
将居中方式改为使用 `margin: 20px auto 0;`，让容器在父元素中水平居中。

**修改文件：** `frontend/src/views/FigureDetail.vue`

**修改前：**
```css
.figure-detail-container {
  margin-top: 20px;
  width: 100%;
  max-width: 1200px;
  margin-left: 50px;
  margin-right: 50px;
  /* ... */
}
```

**修改后：**
```css
.figure-detail-container {
  margin: 20px auto 0;
  width: 100%;
  max-width: 1200px;
  /* ... */
}
```

### 移动端适配
同时优化了移动端响应式样式，减小边距以适配小屏幕设备：

```css
@media (max-width: 768px) {
  .figure-detail-container {
    margin-left: 10px;
    margin-right: 10px;
    padding: 15px;
  }
}
```

## 重构手办服务 - 企业级拆分

### 问题背景
`figure_service.py` 文件行数过多（约610行），包含多种不同职责的业务逻辑，不便于管理和维护。需要进行企业级的拆分。

### 拆分方案

采用 **Facade 模式** 进行服务拆分，将 `FigureService` 拆分为三个子服务，同时保持 `FigureService` 作为统一入口以确保向后兼容。

**架构设计：**
```
FigureService (Facade/门面)
    ├── FigurePriceService (价格计算)
    ├── FigureQueryService (查询筛选)
    └── FigureCrudService (增删改操作)
```

**1. FigurePriceService - 手办价格服务**
- 职责：汇率转换、平均价格计算
- 文件：`figure_price_service.py`
- 主要方法：
  - `convert_to_cny()` - 币种转换为人民币
  - `calculate_order_amount_cny()` - 计算订单金额（人民币）
  - `update_figure_average_purchase_price()` - 更新平均入手价格
  - `calculate_figure_average_purchase_price()` - 计算平均入手价格
  - `calculate_orders_average_price()` - 计算订单平均价格

**2. FigureQueryService - 手办查询服务**
- 职责：查询构建、列表获取、筛选
- 文件：`figure_query_service.py`
- 主要方法：
  - `convert_purchase_type_to_chinese()` - 入手形式转换
  - `build_figure_list_query()` - 构建查询
  - `get_figures_list()` - 获取手办列表
  - `get_figure_by_id()` - 获取手办详情
  - `get_figure_count()` - 获取手办总数
  - `get_figures_with_orders()` - 获取手办及订单

**3. FigureCrudService - 手办CRUD服务**
- 职责：增删改、批量操作
- 文件：`figure_crud_service.py`
- 主要方法：
  - `create_figure()` - 创建手办
  - `update_figure()` - 更新手办
  - `delete_figure()` - 删除手办（软删除）
  - `batch_delete_figures()` - 批量删除手办

**4. FigureService - 手办服务主入口（Facade）**
- 职责：保持向后兼容，作为各子服务的统一入口
- 文件：`figure_service.py`
- 实现方式：所有方法委托给相应的子服务
- 对外接口：保持与原 `FigureService` 完全一致

### 修改文件
- 【新增】`backend/app/services/figure_price_service.py` - 价格服务
- 【新增】`backend/app/services/figure_query_service.py` - 查询服务
- 【新增】`backend/app/services/figure_crud_service.py` - CRUD服务
- 【重构】`backend/app/services/figure_service.py` - Facade模式主入口
- 【更新】`backend/app/services/__init__.py` - 导出子服务

### 优化效果
- ✅ 服务职责单一，每个服务只负责特定领域
- ✅ 代码行数合理，便于管理和维护
  - `figure_price_service.py`: ~170行
  - `figure_query_service.py`: ~180行
  - `figure_crud_service.py`: ~200行
  - `figure_service.py`: ~138行
- ✅ 保持向后兼容，现有调用代码无需修改
- ✅ 便于单元测试，每个服务可独立测试
- ✅ 便于后续扩展，新增功能只需修改对应服务

### 架构优势

**单一职责原则 (SRP)**
每个服务只负责一种类型的业务逻辑，降低耦合度。

**开闭原则 (OCP)**
新增功能时，只需在对应的服务中添加方法，不影响其他服务。

**Facade 模式**
`FigureService` 作为门面，对外提供统一的调用接口，内部实现委托给子服务。

**代码可维护性**
- 文件大小适中，便于阅读和理解
- 功能模块化，便于定位和修复问题
- 便于团队协作，不同开发者可同时修改不同服务

### 部署更新

- 执行 `docker-compose up -d --build` 重新部署服务
